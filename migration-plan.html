Migration Plan: One Nav, One Footer

Collapse the navbar and footer — currently hand-copied into every page and already drifting out of sync — into a single source of truth by extending the existing nav.js. No framework. No build step. No rewrite of page content. Ship incrementally; the site stays live the whole way.

Static HTML Zero build Extends nav.js ~1–2 hrs · low risk
TL;DR for the agent

Create site-chrome.js holding one SITE config (links, dropdown, CTAs, footer) and a render function that injects the navbar + footer into <div id="site-nav"> / <div id="site-footer"> mount points. Active link is derived from location.pathname — no per-page flags. Then, page by page, delete the duplicated <nav>/<footer> markup and drop in the two mount divs. Keep nav.js's behaviour, but call it after render. Validate links, active states, and mobile menu after each page.

Scope: nav + footer chrome only Touches: ~12 root *.html + 2 new/edited JS Excludes: handover/, epub-handoff/, page bodies Rollback: git revert per page

Why

Every page carries three hand-maintained copies of the chrome: the desktop .nav-menu, the .nav-dropdown-menu, and the .nav-mobile list — plus a footer. They have already drifted: the Insights dropdown lists Mathematical Backbone / Social Cognition / Argentina on new pages but not on index.html, model.html, attractors.html, start-here.html. Some links point at pages that don't exist in every deploy. One edit currently means ~12 manual edits; the fix makes it one.

0 Branch & baseline

Make the work reversible and capture a visual baseline before touching anything.

  • Create a branch: git checkout -b chrome-single-source.
  • Build the authoritative page list (grep above). Save it to MIGRATION_PAGES.txt.
  • Screenshot the navbar + footer of index.html and one Insights page at desktop and mobile widths — this is the visual diff target.
  • Note the current active-state convention: class="nav-link active" and class="nav-dropdown-toggle active".

1 Author site-chrome.js — the single source

Keep the exact class names the current markup uses (navbar, nav-container, nav-logo, nav-menu, nav-link, nav-dropdown, nav-dropdown-menu, menu-divider, nav-actions, nav-btn-github, nav-btn-newsletter, nav-hamburger, nav-mobile, footer) so style.css and nav.js keep working untouched.

Add the render function in the same file. It builds the same DOM the pages hand-write today, and marks the active link by comparing each href to the current filename:

function currentPage() {
  const f = location.pathname.split('/').pop() || 'index.html';
  return f === '' ? 'index.html' : f;
}
function renderChrome() {
  const here = currentPage();
  const isActive = (href) => href === here;
  const inDropdown = SITE.insights.items.some(i => i.href === here);

  // ...build innerHTML strings for: nav-menu, dropdown, nav-mobile, footer...
  // add class 'active' to the matching nav-link; if inDropdown,
  // add 'active' to the .nav-dropdown-toggle too (matches current convention).

  const navMount = document.getElementById('site-nav');
  const footMount = document.getElementById('site-footer');
  if (navMount)  navMount.innerHTML  = navHTML;
  if (footMount) footMount.innerHTML = footerHTML;
}
✓ Acceptance — Phase 1
  • renderChrome() called on a scratch page produces markup byte-compatible with an existing page's nav (diff the outerHTML).
  • Active class lands on the correct link for model.html, and on the dropdown toggle for an Insights page.

2 Wire render → behaviour in nav.js

nav.js already handles dropdown toggle + hamburger. It must now run after the chrome is injected, since the elements won't exist until render.

EDIT nav.js
// At the very top of nav.js, wrap the existing IIFE body in initNav()…
function initNav() {
  // ← existing dropdown + hamburger code, unchanged
}
// Boot: render chrome first (if present), then bind behaviour.
if (typeof renderChrome === 'function') renderChrome();
initNav();
Load order matters. On every page, load site-chrome.js before nav.js:
<script src="site-chrome.js"></script> then <script src="nav.js"></script>. Both go at end of <body>, before any page-specific scripts.

3 Convert pages — one at a time

For each file in MIGRATION_PAGES.txt, swap hand-written chrome for mount points. Do one page, verify, commit, repeat — never bulk-edit blindly.

  • Delete the entire <nav class="navbar">…</nav> block. Replace with: <div id="site-nav"></div>
  • Delete the entire <footer class="footer">…</footer> block. Replace with: <div id="site-footer"></div>
  • Ensure the two script tags are present at end of body (Phase 2 order). Most pages already load nav.js — just add site-chrome.js before it.
  • Open the page. Confirm nav + footer render, the correct link is active, the dropdown opens, the hamburger works on mobile.
  • git commit -m "chrome: migrate <page>" — one commit per page for clean rollback.
A <noscript> block with the core links can sit inside #site-nav as a graceful fallback. See Phase 5 for the SEO consideration.

4 Reconcile the link list

Now that links live in one place, make them correct once.

  • Cross-check every href in SITE against files that exist in the real repo. Flag any that 404.
  • Decide the canonical Insights ordering and trailing-link set — this is the version every page now inherits.
  • Remove links to pages that were deleted; add any page missing from the menu.

5 Validate

Prove the migration is invisible to users and complete.

✓ Acceptance — whole migration
  • Visual: nav + footer pixel-match the Phase 0 baseline on desktop and mobile.
  • Active state: every page highlights its own primary link; Insights pages highlight the dropdown toggle.
  • Behaviour: dropdown open/close, outside-click + Escape, hamburger toggle all work (nav.js unchanged).
  • Links: zero 404s from the menu; run a link checker across the built site.
  • No duplication left: grep -rl 'class="navbar"' --include=*.html . | grep -v handover returns only pages that still need migrating (ideally none).
  • Console: no errors on any migrated page.
SEO note (one decision needed). JS-injected nav links may not be seen by some crawlers. The footer links and an up-to-date sitemap.xml keep all pages discoverable, so internal-link equity is preserved via the footer. If nav links must be crawlable, either (a) keep a static <noscript> link list inside #site-nav, or (b) keep the footer chrome static and only de-duplicate the navbar. Recommendation: ship JS nav + static footer link list + sitemap — best effort/value ratio.

6 Land & document

  • Open a PR; include before/after screenshots and the empty-grep proof.
  • Add a short handover/chrome-single-source/HANDOVER.md (matches your existing handover convention) describing how to add a nav link: "edit SITE in site-chrome.js, nothing else."
  • Merge. Future nav/footer changes are a one-line edit.

Out of scope (explicitly)

  • No framework, bundler, or package.json — the site stays openable as flat files.
  • Page bodies, figures, CSV loaders (argentina-figure.js, social-cognition.js, backbone-worked-example.js) are untouched.
  • handover/ and epub-handoff/ snapshots stay frozen.
  • Shared <head>/SEO templating is a separate, optional follow-up — not this plan.
